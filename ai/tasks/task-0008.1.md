# Required Outcomes

## Blocker 1: Already-Complete Claims Must Exit 0

Current failure:

- `performRemoteJobClaim` returns `*jobAlreadyCompleteError` when the backend says `currentStatus: "success"`.
- That error propagates out of `setupRemoteStateHooks`, then `runPlan`, then Cobra/main.
- Duplicate matrix jobs therefore exit non-zero even though "already complete" is a successful remote-state outcome.

Required fix:

- Catch `*jobAlreadyCompleteError` and return `nil` from `runPlan`.
- Keep other remote setup errors as failures.
- Use `errors.As`, not string matching.

Expected shape:

```go
if err := setupRemoteStateHooks(r, plan, planID, execID, backendURL); err != nil {
    var alreadyDone *jobAlreadyCompleteError
    if errors.As(err, &alreadyDone) {
        return nil
    }
    return err
}
```

Add tests proving:

- Already-complete remote `--job` setup returns success.
- The same path does not run the local job.
- Other claim/setup errors still fail.

It is acceptable to test `performRemoteJobClaim` and/or `runPlan` through a small fake backend if that keeps the test focused.

## Blocker 2: `GetRunnable` Must Match Worker JSON

Current failure:

```go
type RunnableResponse struct {
    JobIDs []string `json:"jobIds"`
}
```

Backend contract:

- `packages/types/src/index.ts` has `RunnableJobsResponse { jobs: string[] }`.
- Worker returns the coordinator response as JSON with key `jobs`, not `jobIds`.

Required fix:

- Change the JSON tag to `json:"jobs"`.
- Add or update `internal/remotestate/client_test.go` so a server response like:

  ```json
  { "jobs": ["job-a", "job-b"] }
  ```

  is parsed by `Client.GetRunnable` as `[]string{"job-a", "job-b"}`.
- If you keep backwards compatibility for `jobIds`, do not let it mask the primary contract. The canonical key is `jobs`.

## Blocker 3: `waitForJobRunnable` Must Call `/runnable`

Current failure:

- `waitForJobRunnable` only sleeps.
- It does not call `Client.GetRunnable`.
- The `/runnable` coordination primitive is never exercised.

Required fix:

- Make `waitForJobRunnable` poll backend runnable jobs until `jobID` appears, deadline is exceeded, or context is cancelled.
- Do not rely only on sleeping/backoff.
- Keep the existing claim loop structure: after `waitForJobRunnable` returns because the job is runnable, retry `ClaimJob`.

Recommended implementation:

- Add a small method to `statebackend.Backend`, for example:

  ```go
  RunnableJobs(ctx context.Context, runID string) ([]string, error)
  ```

- Implement it in `RemoteStateBackend` by delegating to `client.GetRunnable(ctx, runID)`.
- Implement it in `FileStateBackend` as a safe local no-op fallback, for example returning `nil, nil`.
- Update `waitForJobRunnable` to call `backend.RunnableJobs`.

Acceptable alternative:

- Define a narrow optional interface used only by `waitForJobRunnable`.
- If you choose this, avoid type assertions to concrete `*RemoteStateBackend` unless there is a good reason.

Tests required:

- When backend returns runnable list without `jobID`, `waitForJobRunnable` waits/polls.
- When backend later returns `jobID`, `waitForJobRunnable` returns and the caller retries claim.
- Context cancellation/deadline exits cleanly.
- `performRemoteJobClaim` with `DepsWaiting` calls the runnable poll path.

Keep tests fast by using short delays/deadlines or injecting a sleep/backoff function if needed.
