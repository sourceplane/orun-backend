- Duplicate matrix runners targeting an already-complete job exit successfully.

Check:

- It does not swallow unrelated errors.
- Tests prove both already-complete and other-error behavior.
- Tests are stronger than only asserting `errors.As` works on a manually created error.

## Blocker 2: Runnable JSON Contract

Required behavior:

- `internal/remotestate.RunnableResponse` parses canonical backend JSON:

  ```json
  { "jobs": ["job-a", "job-b"] }
  ```

- The JSON key is `jobs`, not `jobIds`.
- `Client.GetRunnable` returns the expected slice.
- Test coverage includes non-empty and empty runnable lists.
- The implementation matches `packages/types/src/index.ts` `RunnableJobsResponse`.

## Blocker 3: `waitForJobRunnable` Polls `/runnable`

Required behavior:

- `waitForJobRunnable` calls `backend.RunnableJobs(ctx, runID)`.
- `RemoteStateBackend.RunnableJobs` delegates to `client.GetRunnable`.
- `FileStateBackend.RunnableJobs` is harmless for local code paths.
- The function loops until:
  - target `jobID` appears, then returns nil
  - deadline is exceeded, then returns a timeout error
  - context is cancelled, then returns context error
  - backend runnable call errors, then surfaces the error
- `performRemoteJobClaim` with `DepsWaiting` uses this runnable-poll path before retrying claim.

Check carefully:

- The `delay` parameter is not accidentally ignored in a way that makes tests slow or runtime behavior surprising.
- The function does not sleep before the first runnable call unless this is intentional and acceptable.
- The poll interval and cap are reasonable and documented or obvious.
- Tests do not take excessive real time because of fixed 2s sleeps.
