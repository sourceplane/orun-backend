This task establishes the authentication boundary required before the Worker API can safely expose coordination, logs, account, and dashboard endpoints.


The module must support:

- Orun session JWT issuance and verification using HMAC-SHA256
- GitHub OAuth helper flow for UI login
- Deploy-token bootstrap auth
- Namespace extraction and lazy namespace slug upsert

The key architectural boundary: this task implements authentication helpers and tests. It must not implement the full Worker API router, run/job handlers, rate limiting, account CRUD, or remote-state client behavior. Task 0006 will wire these helpers into endpoint routing.

## GitHub OIDC Verification

Implement:

```typescript
export async function verifyOIDCToken(token: string, env: Env): Promise<OIDCClaims>;
export function extractNamespaceFromOIDC(claims: OIDCClaims): Namespace;
export function looksLikeOIDC(token: string): boolean;
```

Required verification:

- Fetch JWKS from `env.GITHUB_JWKS_URL`.
- Cache JWKS in memory for 15 minutes by URL.
- Decode header and require `alg === "RS256"` and a non-empty `kid`.
- Find matching JWK by `kid`.
- Import the JWK with Web Crypto and verify the RSASSA-PKCS1-v1_5 SHA-256 signature.
- Verify `iss === "https://token.actions.githubusercontent.com"`.
- Verify `aud === env.GITHUB_OIDC_AUDIENCE` exactly. If `aud` is an array, it must contain the exact audience.
- Verify `exp > now` with no grace period.
- Verify `iat <= now + 60`.
- Require non-empty claims:
  - `repository`
  - `repository_id`
  - `repository_owner`
  - `repository_owner_id`
  - `actor`
- Return typed `OIDCClaims`.

Testing requirements:

- Generate an RSA key pair with Web Crypto in tests.
- Sign a valid RS256 test token and verify it.
- Test invalid signature, wrong issuer, wrong audience, expired token, future `iat`, missing required claims, unknown `kid`, malformed JWT, unsupported `alg`, and JWKS cache reuse.
- Mock `fetch` for JWKS. Do not call the real GitHub JWKS endpoint in unit tests.

## Request Authentication

Implement:

```typescript
export type RequestContext =
  | {
      type: "oidc";
      namespace: Namespace;
      allowedNamespaceIds: string[];
      actor: string;
    }
  | {
      type: "session";
      namespace: null;
      allowedNamespaceIds: string[];
      actor: string;
    }
  | {
      type: "deploy";
      namespace: null;
      allowedNamespaceIds: ["*"];
      actor: "system";
    };

export async function authenticate(
  request: Request,
  env: Env,
  ctx?: Pick<ExecutionContext, "waitUntil">,
): Promise<RequestContext>;
```

Required behavior:

- If `X-Orun-Deploy-Token` is present:
  - Require `env.ORUN_DEPLOY_TOKEN`.
  - Compare exactly.
  - Return deploy context only on match.
  - Reject mismatch with `UNAUTHORIZED`.
- Require `Authorization: Bearer <token>` for OIDC/session auth.
- Detect OIDC by decoding the payload issuer without trusting the token.
- Verify OIDC tokens with `verifyOIDCToken`.
- Extract namespace from OIDC using `repository_id` as canonical namespace ID and `repository` as display slug.
- Lazily upsert namespace slug into D1 on OIDC auth.
  - If `ctx?.waitUntil` is present, use it.
  - If not present, await the upsert so tests and non-Worker callers remain deterministic.
- Verify session tokens with `verifySessionToken`.
- Return session context with `namespace: null` and `allowedNamespaceIds`.
- Do not make authorization decisions beyond identity extraction in this task. Namespace access checks belong to the Worker routing task.

Testing requirements:

- OIDC request returns OIDC context and upserts namespace.
- Session request returns session context.
- Deploy-token request returns deploy context.
- Missing auth returns `UNAUTHORIZED`.
- Invalid deploy token returns `UNAUTHORIZED`.
- Invalid OIDC/session token returns `UNAUTHORIZED`.
- OIDC slug upsert is idempotent and updates changed slugs.
