# Required Outcomes

## 1. Browser OAuth Return Flow

Extend GitHub OAuth without breaking the existing API-client JSON callback.

Required behavior:

- `GET /v1/auth/github` continues to work with no query parameters.
- `GET /v1/auth/github?returnTo=<url>` starts OAuth for the dashboard.
- The signed OAuth `state` must bind any `returnTo` value so it cannot be altered during callback.
- `returnTo` must be validated before it is accepted.
- Preferred validation:
  - Add `ORUN_DASHBOARD_URL?: string` to the Worker env contract.
  - If `ORUN_DASHBOARD_URL` is configured, only allow `returnTo` URLs with the same origin.
  - If it is not configured, allow same-origin callback URLs only for local/dev compatibility.
  - Reject malformed or disallowed URLs with a typed `INVALID_REQUEST` JSON error.
- `GET /v1/auth/github/callback` keeps returning JSON when no dashboard `returnTo` is present.
- When dashboard `returnTo` is present, the callback returns a `302` redirect to that URL with the session payload in the URL fragment.

Suggested fragment fields:

```text
#sessionToken=<urlencoded-token>&githubLogin=<urlencoded-login>&allowedNamespaceIds=<urlencoded-json-array>
```

Security requirements:

- Never put the session token in the query string.
- Never include the GitHub OAuth access token in the redirect, response body, logs, D1, R2, or session JWT.
- Do not persist GitHub OAuth access tokens.
- Do not add cookies unless you fully test the CORS and same-site semantics. The current API contract is bearer-token based.
- Preserve existing OAuth tests and add focused tests for:
  - no `returnTo` keeps JSON response
  - valid dashboard `returnTo` redirects with a fragment
  - disallowed `returnTo` is rejected
  - tampered state is rejected
  - GitHub access token is not present in redirect URL

# Integration Notes

- `GET /v1/runs/:runId` may return a `Partial<Run>` from coordinator-backed state. The client and dashboard must tolerate missing dashboard-index fields.
- Session reads are D1-backed and can lag behind the coordinator. The UI should present refresh states clearly.
- Some jobs may have `logRef: null` even when logs exist or vice versa due to async mirrors. A failed/missing log fetch should not crash the run detail view.
- `GET /v1/accounts/me` returns `NOT_FOUND` until the account is created. The dashboard should handle this as an empty account state, not a fatal auth failure.
- `GET /v1/accounts/repos` returns `{ repos: [] }` when no account exists.
