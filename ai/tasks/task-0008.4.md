# Required Outcomes

## 1. Local State Lock Design

Add a real cross-process local state lock around local execution state mutations.

Requirements:

- The lock must be scoped to an execution ID, not global to the whole repo.
- The lock file should live under the execution directory, for example:

  ```text
  .orun/executions/<exec-id>/.lock
  ```

- Lock acquisition must be bounded and cancellable or have a clear timeout.
- Lock release must be reliable on normal exit.
- Stale lock behavior must be thought through and tested or documented.
- Use atomic write/rename for state and metadata writes where needed.
- Do not introduce a daemon, background service, database, or network dependency.
- Prefer standard library or a tiny well-maintained dependency if cross-platform file locking requires one. If you add a dependency, justify it in the report.

## 2. Local Claim Semantics

Update local behavior so concurrent local runs sharing an execution ID have coherent job semantics.

Required semantics:

- If a job is already `completed` or `success`, a second process should exit 0 for that job.
- If a job is `running`, a duplicate local process should not execute the same job concurrently.
- If a dependency failed, downstream jobs should fail/skip cleanly rather than running.
- If dependencies are pending/running, local behavior should either wait with backoff or return a clear dependency-wait message. Prefer waiting if it can be done without surprising existing users.
- State files must remain valid JSON after concurrent runs.
- Metadata counters must not double-count completed/failed jobs.
- Existing single-process local behavior must remain compatible.

You may implement these semantics in `state.Store`, `FileStateBackend`, `runner.Runner`, or a narrow adapter layer after inspecting the current architecture. Keep the change cohesive and tested.

## 3. Tests

Add focused tests that prove the local lock behavior.

Minimum coverage:

- Two concurrent claim/update attempts for the same job cannot both execute the job.
- Duplicate already-complete local job exits cleanly.
- Failed dependency blocks or fails downstream local execution coherently.
- State JSON remains parseable after concurrent local updates.
- Lock acquisition timeout/cancellation path returns a useful error.
- Existing local `orun run`, `orun status`, and `orun logs` tests still pass.

Use deterministic in-process tests where possible. If an end-to-end test needs subprocesses, keep it small and skip only when there is a concrete platform limitation.
