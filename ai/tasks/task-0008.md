## Remote Run Flow

Remote `orun run` must:

1. Resolve or generate the plan using existing behavior.
2. Compute `planID` with `state.PlanChecksumShort(plan)`.
3. Derive `runID`.
4. Initialize or join the backend run with `POST /v1/runs`.
5. Use the existing runner/executor for local step execution.
6. Coordinate selected jobs through backend claim/update/heartbeat/log APIs.

Remote `--job` is critical:

- It must not fail just because local state is missing dependency completions.
- It should ask the backend to claim the job.
- If the backend says dependencies are waiting, poll `/v1/runs/{runID}/runnable` with exponential backoff and retry claim.
- If dependencies are blocked or failed, exit non-zero with a clear upstream dependency message.
- If another runner already completed the job, exit 0.
- If another runner is running the job, poll status/backoff until the job completes, fails, or becomes claimable.

Suggested claim behavior mapping:

| Backend response | CLI behavior |
| --- | --- |
| `claimed: true` | Execute the job. |
| `claimed: true, takeover: true` | Execute and mention takeover in verbose/GHA output. |
| `claimed: false, currentStatus: "running"` | Poll and retry or exit 0 when completed by another runner. |
| `claimed: false, currentStatus: "success"` | Treat as already complete and exit 0 for that job. |
| `claimed: false, currentStatus: "failed"` | Exit 1. |
| `claimed: false, depsWaiting: true` | Poll `/runnable`, then retry claim. |
| `claimed: false, depsBlocked: true` | Exit 1 with dependency-blocked message. |

Use exponential backoff with jitter:

- Start at 2 seconds.
- Cap at 60 seconds.
- Default dependency wait timeout: 30 minutes.
- It is acceptable to make this timeout a constant for now if you document it.

Start a heartbeat every 30 seconds while a remote-claimed job is executing. Stop the heartbeat when the job reaches a terminal state.

Terminal updates:

- On success, send `status: "success"` with `runnerId`.
- On failure, send `status: "failed"`, `runnerId`, and concise error text.
- Ensure final update is attempted even when a step fails.

Logs:

- Existing local logs are step-level, but the backend is job-level.
- Aggregate remote job logs with clear step section headers, or upload the full accumulated job log after each step.
- `orun logs --remote-state --job <jobID>` should display the backend's job-level log.
