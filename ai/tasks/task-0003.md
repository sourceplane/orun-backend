# Endpoint Behavior

## `POST /init`

Body:

```typescript
{ plan: Plan; runId: string; namespaceId: string; namespaceSlug: string }
```

Behavior:

- Idempotent for the same `runId`.
- If state already exists with the same `runId`, return `{ ok: true, alreadyExists: true }`.
- If state already exists for a different `runId`, return a conflict.
- Initialize all plan jobs as `pending`.
- Set `runnerId`, `startedAt`, `finishedAt`, `lastError`, and `heartbeatAt` to `null`.
- Set run status to `running`.
- Set `createdAt` and `updatedAt` to current ISO timestamp.
- Persist under key `runState`.
- Return `{ ok: true, alreadyExists: false }`.

Validate at minimum:

- `plan.jobs` is an array.
- Every plan job has a non-empty `jobId`.
- Duplicate `jobId`s are rejected.
- Dependencies refer to jobs that exist in the same plan.

## `POST /jobs/:jobId/claim`

Body:

```typescript
{ runnerId: string }
```

Behavior:

- If job is `pending` and all dependencies are `success`, mark it `running`, set `runnerId`, `startedAt`, and `heartbeatAt` to now, persist, and return `{ claimed: true }`.
- If job is `pending` and any dependency is `failed`, return `{ claimed: false, currentStatus: "pending", depsBlocked: true }`.
- If job is `pending` and any dependency is `pending` or `running`, return `{ claimed: false, currentStatus: "pending", depsWaiting: true }`.
- If job is `running` and `heartbeatAt` is older than 300 seconds, transfer ownership to the new runner, update `heartbeatAt`, persist, and return `{ claimed: true, takeover: true }`.
- If job is `running` and heartbeat is still fresh, return `{ claimed: false, currentStatus: "running" }`.
- If job is terminal, return `{ claimed: false, currentStatus: job.status }`.

Spec note:

`spec/05-coordinator-do.md` includes `depsBlocked` and `depsWaiting`, but `ClaimResult` from `@orun/types` does not currently include those flags. Handle this deliberately. Preferred approach: define a package-local coordinator response type that extends `ClaimResult` with optional `depsBlocked` and `depsWaiting`, then document the contract gap in the implementer report. Only change `@orun/types` if you can justify it as a small backwards-compatible contract fix and update tests.

## `POST /jobs/:jobId/update`

Body:

```typescript
{ runnerId: string; status: "success" | "failed"; error?: string }
```

Behavior:

- Validate `runnerId`.
- Reject update if `job.runnerId !== runnerId`.
- Reject update if the job is not currently `running`.
- Set terminal status, `finishedAt`, and `lastError`.
- If all jobs are `success`, set run status to `completed`.
- If any job is `failed`, set run status to `failed`.
- Always update `runState.updatedAt`.
- Persist.
- If run becomes `completed` or `failed`, schedule expiry alarm 24 hours in the future.
- Return `{ ok: true }`.

Spec note:

`UpdateJobRequest` in `@orun/types` does not include `runnerId`, while the coordinator spec requires runner ownership verification. Preferred approach: define a package-local `CoordinatorUpdateJobRequest = UpdateJobRequest & { runnerId: string }`. Do not remove the ownership check.

## `POST /jobs/:jobId/heartbeat`

Body:

```typescript
{ runnerId: string }
```

Behavior:

- If `job.runnerId !== runnerId`, return `{ ok: false, abort: true }`.
- If `job.status !== "running"`, return `{ ok: false, abort: true }`.
- Otherwise update `heartbeatAt` to now, update `runState.updatedAt`, persist, and return `{ ok: true }`.

## `GET /jobs/:jobId/status`

Return the current job state:

```typescript
{
  jobId: string;
  component: string;
  status: JobStatus;
  deps: string[];
  runnerId: string | null;
  startedAt: string | null;
  finishedAt: string | null;
  lastError: string | null;
  heartbeatAt: string | null;
}
```

## `GET /runnable`

Return:

```typescript
{ jobs: string[] }
```

Only include jobs where:

- `status === "pending"`
- all dependencies have status `success`

Jobs blocked by failed dependencies must not be returned.

## `GET /state`

Return the full `RunState`.

This is used later by the Worker to update D1-derived dashboard rows. It must not read from D1 or R2.

## `POST /cancel`

Behavior:

- Set all `pending` and `running` jobs to `failed`.
- Set their `lastError` to `"cancelled"`.
- Set `finishedAt` for affected jobs if it is null.
- Set run status to `cancelled`.
- Update `runState.updatedAt`.
- Persist.
- Schedule expiry alarm 24 hours in the future.
- Return `{ ok: true }`.

# Alarm / Garbage Collection
Implement:

```typescript
async alarm(): Promise<void>
```

Behavior:

- `scheduleExpiry()` sets an alarm only if one is not already set.
- Expiry is 24 hours after terminal state.
- `alarm()` deletes all Durable Object storage via `this.state.storage.deleteAll()`.
- Clear the in-memory `runState` after delete so subsequent `/state` behaves as uninitialized.
