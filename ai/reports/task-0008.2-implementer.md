## Cloudflare Worker Contract

The `cloudflare-worker-turbo` and `cloudflare-worker` compositions now support:

| Feature | Input | Purpose |
| --- | --- | --- |
| Wrangler config flexibility | `wranglerConfig` | Accept `wrangler.jsonc` or `wrangler.toml` (auto-detected by default) |
| Resource provisioning | `preDeployCommand` | Bootstrap D1 databases, R2 buckets, etc. before deploy |
| D1 migrations | `migrationCommand` | Apply D1 migrations as a first-class deploy step |
| Deploy override | `deployCommand` | Override the default `pnpm run deploy` |
| Live verification | `smokeCommand` | Verify the live endpoint post-deploy |

Deploy step now emits GitHub Actions annotations:
- `::notice::DRY-RUN:` — dev environment dry-run
- `::notice::SKIPPED:` — not production branch/environment
- `::notice::LIVE DEPLOY:` — actual deployment beginning
- `::notice::DEPLOY SUCCESS:` — deployment completed

Credentials (`CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_API_TOKEN`) are validated before any deploy attempt.

## Production Push Semantics

Created `docs/production-deploys.md` documenting:
1. The `--changed` trap (squash merges → zero jobs → false green)
2. Three recommended patterns:
   - Pattern 1: Run all production jobs on main (recommended for small repos)
   - Pattern 2: `--changed` with explicit `--base ${{ github.event.before }}`
   - Pattern 3: Separate deploy workflow on main
3. How to verify a deploy actually happened (check for CI annotations)
4. Consumer repo checklist for production readiness
