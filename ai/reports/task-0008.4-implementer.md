# Task 0008.4 Implementer Report

## Summary

Added cross-process local state locking to `sourceplane/orun` using POSIX `flock(2)` advisory file locks. The `FileStateBackend` now provides real claim semantics — when multiple `orun run` processes share the same `--exec-id`, only one can claim and execute each job. The implementation uses the existing `RunnerHooks` pattern (already proven for remote state) to wire lock-aware claim/update logic into local execution without any changes to `runner.go` or the `Backend` interface.

## Local Lock Design

- **Mechanism**: `flock(2)` (POSIX advisory file lock) via `golang.org/x/sys/unix` (already a direct dependency at v0.43.0)
- **Scope**: One lock per execution ID at `.orun/executions/<exec-id>/.lock`
- **Acquisition**: Non-blocking attempt (`LOCK_EX|LOCK_NB`) with 50ms retry loop, respecting `ctx.Done()` for bounded timeout
- **Stale lock handling**: Advisory locks auto-release when the file descriptor is closed or the process exits/crashes. No manual cleanup required.
- **Platform**: Build-tagged `flock_unix.go` for darwin+linux, `flock_other.go` (no-op) for other platforms
- **No new dependencies added**

## Local Claim Semantics

`ClaimJob` now performs these checks under the file lock:

1. **Already completed** → returns `Claimed=false, CurrentStatus="completed"` → runner skips
2. **Already running** → returns `Claimed=false, CurrentStatus="running"` → runner skips
3. **Already failed** → returns `Claimed=false, CurrentStatus="failed"` → runner errors
4. **Dependencies pending/running** → returns `DepsWaiting=true` → runner errors
5. **Dependencies failed** → returns `DepsBlocked=true` → runner errors
6. **Claimable** → marks `running`, writes state atomically → runner executes

`UpdateJob` persists terminal status (`completed`/`failed`) under the same file lock, acting as the cross-process synchronization point.

`RunnableJobs` returns job IDs whose dependencies are all `completed` based on current state.

## Assumptions

1. **Single-process local runs remain unaffected**: The `setupLocalStateHooks` path is only wired for non-dry-run mode. The `BeforeJob` hook adds negligible overhead (one lock+read+write per job) and the test suite confirms backward compatibility.
2. **Stale "running" jobs after crash**: If a process crashes after claiming but before terminal update, the job stays `running`. This is acceptable — `--retry` resets and re-runs. Future work could detect stale runners via PID recording.
3. **Step-level updates don't need the file lock**: Only the claiming process writes step progress. Cross-process coordination is only needed at job boundaries (claim + terminal).

## Remaining Gaps

1. **Remote-state conformance**: The conformance workflow (`examples/remote-state-matrix/intent.yaml`) still uses the outdated format (missing `compositions.sources`). This is a Task 0008.5 or fixture-update scope item, not a local-lock concern.
2. **Dependency waiting with backoff**: Currently the local `BeforeJob` hook returns an error on `DepsWaiting` rather than polling. For the default single-process `orun run` (which schedules jobs topologically), this is never hit. For multi-process usage with `--job`, the caller should retry. Adding poll-and-wait semantics is a potential follow-up.
3. **Windows support**: The `flock_other.go` stub provides no-op locking on Windows. True cross-process safety on Windows would require `LockFileEx` — out of scope for this task.
