## Issues

### Issue 1: Execute command deviates from spec (non-blocking)

**Spec** (`spec/02-devops.md`): `kiox -- orun run --execute --gha`  
**Workflow** (`.github/workflows/workflow.yml:44`): `kiox -- orun run`

The `--execute` flag appears to be orun's default behavior (`orun run` executes by default). The `--gha` flag appears auto-detected from the GitHub Actions environment (CI log confirmed `gha` scope in Execute output). The execution ran correctly: 15 jobs, all green, no unintended deploy.

This is a spec-fidelity gap, not a behavioral failure. Recommend aligning to `kiox -- orun run --execute --gha` in a follow-up to be explicit and future-proof.

### Issue 2: `moduleResolution: Bundler` instead of spec's `NodeNext` (justified deviation)

**Spec** (`spec/01-monorepo-structure.md`): `module: NodeNext`  
**Implementation** (`tsconfig.base.json`): `moduleResolution: Bundler`

Justified: Cloudflare Workers require Bundler resolution. `Bundler` also enables source-level type resolution across workspace packages without a prior build step, which is required for `turbo run typecheck` with `dependsOn: ["^typecheck"]`. Noted in implementer report. Not a blocker for scaffold.

### Issue 3: `.npmrc` is empty

The `.npmrc` file exists with zero content. The implementer report describes it as approving esbuild/workerd build scripts, but pnpm 10.12.1 requires interactive `pnpm approve-builds` for that. Local `pnpm install` still warns about ignored build scripts. This does not affect typecheck, build, or CI (kiox-action handles workspace setup in CI). Low risk for scaffold stage.

### Issue 4: `kiox.lock` not committed

`kiox.lock` (generated by `kiox -- orun plan`) is present locally but not committed to the PR and not listed in `.gitignore`. It pins the exact OCI digest of the orun provider (`sha256:c259e8c8f2ebacbe68a7eccbbb52fe29e5caa7a502243ef4cc6146664896e1a6`). Lock files improve reproducibility. Recommend committing `kiox.lock` or explicitly adding it to `.gitignore` in a follow-up.
